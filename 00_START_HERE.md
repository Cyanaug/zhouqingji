# 00 · START HERE —— 昼青集 · 读诗剧场

> 给实现方 agent（CC / Fable 5）的总入口。先读这份，再读 01–04。

## 这是什么

一个**本地优先**的应用：让许多 AI 读者（不同模型 × 不同读者背景）阅读作者 cyan 的诗，各自打分、写下反应；作者能看见「一首诗在很多眼睛里的形状」。

**北极星（一句话，别偏离它）**：项目要兑现作者一首诗《夜路》的结尾——「由世间所有的所见将它命名」。所以核心价值不是"评出最好的诗"，而是"让每首诗被众多真诚的观看层层照亮"。任何设计决策拿不准时，回到这句。

## 本包文档

- `01_corpus_schema.md` —— 作品总集（诗稿）的数据结构。**冻结**。
- `02_readers_and_casting.md` —— 读者模型、选角层、读者底线。底线**冻结**，具体人设**你来设计**。
- `03_runner_and_coverage.md` —— 阅读记录结构、传输层、覆盖账。结构**冻结**，传输实现**你来选**。
- `04_app_and_design.md` —— 信息架构与视觉定调。架构**冻结**，视觉/交互**你来发挥**。

## FROZEN vs LATITUDE（最重要的协作原则）

- **FROZEN（冻结，不要改）**：两张 schema（作品总集、阅读记录）的字段、读者底线、榜单不得由 LLM 排名、人改↔AI跑 的 content_hash 契约、v1 范围。这些改动会造成数据迁移或价值漂移，代价高。
- **LATITUDE（放手去做，鼓励你改进）**：视觉系统、页面细节、具体读者人设、传输层怎么实现、分布图长什么样、交互微观体验。**这些地方你（Fable 5）比文档更该有判断，请主动优化，并在实现时把你的改动和理由记在 `theater/NOTES.md` 里供作者复核。**

一句话：**骨头定死，器官可换。** 你在器官上尽情发挥。

## INITIALIZATION（你启动时要做的第一件事）

作者只会：建一个空的 `昼青集\` 目录，把本包全部文档 + 诗稿 JSON 丢进去，然后说一句"照这些搭"。**目录、放文件、分层、写说明，全部由你完成，不要让作者手动整理。** 具体：

1. 在 `昼青集\` 下建出下面的结构；
2. 把作者丢进来的 `诗稿*.json` / `poems_*.json` 归入 `corpus\`，原始导出归入 `corpus\raw\`，`昼青·诠释.md` 归入 `corpus\`；
3. 写一份 `README.md`（说明三层各是什么 + 下面这条边界）；
4. 一切代码只在 `theater\` 里写。

```
昼青集\                         # 人看：中文根目录
├─ corpus\                     # 资产：只进不毁、作者所有、可回滚
│  ├─ 诗稿.json                 # 作品总集（见 01）；由现有 poems_*.json 合并而来
│  ├─ 昼青·诠释.md              # 读解档案，作者手工增删
│  └─ raw\                     # 原始导出留底（按来源分子目录，如 device-a\ / device-b\）
├─ theater\                    # 机器跑：本应用的代码，可重写可弃
│  ├─ src\  runners\  personas\  NOTES.md
└─ results\                    # 产出：阅读记录，随时间累积、永不覆盖
   └─ reads\...
```

**硬边界（写进 README 第一行）**：`theater` 里的代码**读 `corpus`、写 `results`，永远不修改 `corpus` 内的作品内容**。清洗、删改、剪自注、改可见性，只由作者在 GUI 里做。

## BUILD ORDER（先做什么）

- **v1（先跑起来、先看见"形状"）**：01–04 的地基 + 最小闭环——盲读 → 诗详情页（诗 + 读者反应 + 反应分布图）+ 时间轴视图 + 一个招牌榜。这一版就兑现了北极星的九成。
- **v2（想清楚再上，schema 已留钩子，纯加模式不迁移）**：论坛式对话读（thread）、AI 选角层自动差异化、时序感悟分析、知情/初见对照视图。

**不要在 v1 里做论坛。** 论坛是 v1 的糖、v2 的骨。先让作者看见形状。
