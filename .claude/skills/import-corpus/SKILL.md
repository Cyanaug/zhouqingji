# 原始素材 → corpus/诗稿.json

这份指南不认格式——不管原始素材是 Word、txt、表格还是别的什么，思路是一样的：先看懂目标 schema 和现有先例，再动手，遇到判断不了的地方停下来问，不要凭通用规则批量硬做。

## 0. 先看这三样，别的不用我教

1. **`01_corpus_schema.md`**——`诗稿.json` 每个字段的唯一权威定义，FROZEN，照这个来。
2. **现有 `corpus/诗稿.json`**（如果已经有内容）——活样本，看真实数据长什么样。
3. **`theater/src/build_corpus_huawei.py` / `merge_corpus_xiaomi.py`**——两个先例脚本，`sha1()` 计算、`id` 续编号、备份再写这几个具体做法直接抄；但它们是设备专属的历史脚本，**结构抄、参数别抄**。

## 1. 第一步：判断新建还是追加

**先检查 `corpus/诗稿.json` 是否已存在**，这决定走哪条分支——两条分支的安全阀不一样，走错等于白做：

- **不存在（新建）**：一次性全量生成、一次性写入。写之前挑一个合理的 ID 前缀方案（比如从这批作品的作者笔名/项目名衍生一个短前缀），**不用专门开口问用户"想要什么前缀"——默认自己定一个，写完提一句"用了 xxx- 编号，想换的话说一声"就够**，多数人不在乎，在乎的人自己会说。真动手前也确认一遍文件确实不存在（防止判断错分支）。
- **已存在（追加）**：先在 `corpus/.backups/` 落一份带时间戳的备份（照抄 `merge_corpus_xiaomi.py` 的做法），读现有条目**确定当前用的编号方案**（不要假设是 `zq-`，谁都可能改成别的前缀，续着现有的来就行）取最大编号接着编，**只加、不改、不删已有条目**。

## 2. 硬规则——这些事不能自动做，会踩本项目已经吃过的亏

- **分界模式必须先问，不能凭"空行/标题样式"批量猜**。空行在诗里既可能是"两首诗之间"也可能是"同一首诗换节"——这个项目自己的 app.js 里就写着"双倍行距噪音和真分段长得一模一样，猜不出来，猜错了还会悄悄吞掉真分段"，华为/小米两批真实数据都栽过这个坑。做法：先切 3~5 个样本给用户看，确认这份素材里"分界"到底长什么样（空行？Word 标题样式？手打的分隔符？分页符？），确认了模式再批量处理。
- **content/note 不自动拆分**。`01_corpus_schema.md` 里这是 FROZEN 立场："作者不预先清洗……GUI 提供划取一段移入 note 的动作"——自动识别自注是故意不做的事。全部先进 `content`，`note` 留空，导入完提醒用户去 GUI 里手动划一次自注（如果有诗正文后面跟着大段自评/背景说明）。
- **不写自动去重**。`01_corpus_schema.md`"后续"里写明了"不写去重脚本，个别重复由作者手工确认合并"——发现疑似重复（标题/内容高度相似）只报告给用户，不自动跳过也不自动合并。
- **`date_written` 优先诗自己的落款**：诗末尾/开头如果自带写作时间，抄一份进这个字段（原文不删，落款本来就是内容的一部分）；没有落款就留 `null`。Word 文件自带的创建/修改时间戳（`core_properties.created`）或者文件系统 mtime **不能拿来冒充写作时间**——那记的是"这份 Word 文件什么时候存的"，跟"这首诗什么时候写的"经常对不上，宁可留空也别拿假数据充数。
- **`guid`/`created`/`modified` 是设备专属字段**，本地文件没有这些就留空，不要编。
- **`content_hash` = `sha1(content)`**（两个先例脚本都有现成实现），且必须用**最终要存进 `content` 字段的那份文本**去算——不是清洗前的草稿版本，否则以后作者一编辑就会被误判成"改过"。
- **`genre`/`ai_read`/`visibility` 别无脑全填诗的默认值**：混进来的杂文/日记这类，`genre` 该标"杂文"、`ai_read` 该给 `false`（不进 AI 读者池）；拿不准就问用户，不要自己猜体裁。

## 3. Word 文档额外的技术坑

- **软换行（Shift+Enter）容易丢**：多数"读段落文本"的路数只会保留真正的段落分隔（Enter），诗句内部靠 Shift+Enter 换行的部分会被悄悄拼成一整行——对诗是致命的，格式上完全看不出来。写完一定要**抽查几首、逐行比对原文行数**；更保险的路子是请用户自己在 Word 里"另存为纯文本"，用那份结果核对行数是否对得上。
- **表格、图片、分页符**：遇到就停下来问用户怎么处理，不要自动跳过或瞎猜降级处理。分页符本身可以是"下一首开始"的一种分界信号，可以列入第 2 节"先问分界模式"时的候选依据之一。

## 4. 落盘前的验证清单

1. 生成完整的条目列表后，先跑一遍字段完整性检查（对照 `01_corpus_schema.md` 必需字段，别漏字段）。
2. **一次性写，不要边生成边写**——生成完整、检查完整再落盘；追加分支要确认第 1 步的备份已经落地。
3. 写完抽查 3~5 条：启动 `python theater/src/server.py`，浏览器里实际打开看效果——标题/正文拆得对不对、有没有乱码、`content_hash` 对不对得上。
4. 收尾提醒用户：去 GUI 里手动划一遍自注（如果有）、确认一遍疑似重复条目、确认一遍 genre/visibility 分类。
