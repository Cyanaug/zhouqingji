# 02 · 读者与选角 Readers & Casting

## 读者模型

一个"读者"是一个配置：

```json
{
  "persona_id": "classical-scholar",
  "model": "claude-... | gpt-... | gemini-... | ...",
  "persona": "老学究，重格律与用典，读诗先看结构",
  "knows_诠释": false,      // 是否预先读过 昼青·诠释.md（知情 vs 初见）
  "knows_date": false,     // 是否知道写作时间（时序感 vs 时间盲）
  "reads_background": true  // 是否读该诗的 background 小注
}
```

"多样性"来自两根轴，别混：**真·不同模型**（Claude/GPT/Gemini/DeepSeek…）+ **读者背景**（人设）。第一版只用一个模型 × 十来种人设即可跑出八成效果。

## 选角层 Casting（LATITUDE：具体人设由你设计）

不要让作者手搓读者。设一个 **casting 层**，给它一组轴，由它实例化读者：

- 年代/世代、母语（含母语非中文者）、感受取向（重感觉 / 重结构 / 重形式 / 重语言游戏）、`knows_诠释`、`knows_date`。
- 进阶（v2）：**"看诗选角"**——挑那些大概率会对某首诗产生分歧的读者来读它。
- subagent 下发时，人设就已经差异化好；作者不必参与造读者，最多跟 agent 一句话调轴。

**你（Fable 5）负责设计这组人设的具体清单，并写进 `theater/personas/`；把设计理由记进 `NOTES.md`。**

## 读者底线 —— FROZEN（这是全项目最要紧的规则）

全体读者唯一不可动摇的**公约数：都愿意认真读一首诗**。守住它，多样即财富；破了它（塞入不读诗、只想贬低的人设），就是噪声，不是多样性。

具体刻死几条：

1. **读懂并说出诗里的感受，是好读者。** 关于 craft（意象、结构、语言、节奏）的逆耳批评要保留——读者不是只会夸的捧场客，作者要听真话。
2. **但"情绪低沉 = 诗差"是误读，必须拦掉。** 有些诗很消沉；因为它低沉就判它"不好"，是读者失职，不是诚实。低沉是被读懂的对象，不是被扣分的理由。在读者系统提示里明确区分这两者。
3. **需要语境才能读的诗**：若该诗有 `background` 小注且读者 `reads_background=true`，让它在知道那个人、那段处境的前提下读——读到的是一个人，而不只是一页纸。最私人、暂不想被这样读的诗，作者会设 `private`，根本不进读者池，你不必处理。
4. **不邀请踩踏。** 永远不要给读者"挑出最差/最离谱的诗"这类任务（会诱导表演、迎合、贬低）。读者只管认真读**它面前的这一首**，给真实反应。所有榜单都是事后从这些反应里派生的（见 04）。

## 读法模式 context_mode（决定读者能否看到别人的评论）

两种，**不能混进同一个指标**：

- **`blind`（盲读）**：读者上下文里没有别人的评论，每次都是独立的一次阅读。→ 用来测诗的真实"形状/方差"。**成本线性**（一诗 × 一人设），可无限累积不涨。**这是积累的主体。**
- **`thread`（对话读，v2）**：读者看得到整个帖子、能顶能反驳。→ 热闹、像论坛，但必然收敛、互相带偏。**成本随楼层膨胀**，故只对精选的少数诗（招牌/最两极那几首）定点开、且给楼层上限。**这是偶尔办的沙龙，不是日常。**

作者的顾虑"重读累积会不会带偏、导致不分化"——只发生在 thread；blind 永远独立。分开存、分开统计。

## 知情/时序两个开关

- `knows_诠释`：知情读者（读过诠释册）vs 初见读者（纯直觉）。两种读法的差异本身是好看的光谱。
- `knows_date`：知道写作时间的读者能顺着"这首在你体系里偏移了"这类只有对比才见的东西去读；时间盲读者每首当孤篇。都保留。作者那句"AI 读多了会不会注意到时间轴"——正是 `knows_date=true` 读者该干的事。
