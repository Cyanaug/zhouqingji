# 01 · 作品总集 Schema（诗稿.json）—— FROZEN

## 形态

**单一 JSON 文件** `corpus/诗稿.json`，一个数组，每条一部作品，`id` 为主键。不按设备分文件（设备是"出处"不是"分类"）。所谓"现代诗/词/榜单"都是 app 读取时按字段筛出的**视图**，不是物理文件。

由你自己的原始导出（例如 `poems_modern.json` / `poems_ci.json` / `prose_backup.json` 之类的多个来源文件）合并生成，入库时给每条发一个**独立于设备**的内部 `id`。

## 每条的字段

```json
{
  "id": "zq-0001",                     // 内部主键，入库时生成，设备无关
  "author": "your-pen-name",           // 现单人；多作者时按此过滤，勿删此字段
  "title": "示例：无题",
  "genre": "现代诗",                    // 现代诗 | 词 | 歌词 | 杂文（可后期重标）
  "content": "这里是这首诗的正文……\n...", // 正文纯文本；默认=全文
  "note": "",                          // 自注；默认空。作者在 GUI 里从 content 剪入
  "date_written": "2022-06",           // 诗中显式标明的写作时间；无则 null
  "created": "2022-06-xxT..",          // 备忘录自带创建时间戳
  "modified": "..",                    // 备忘录自带修改时间戳
  "visibility": "public",              // public | private，默认 public
  "ai_read": true,                     // 是否进入 AI 读者池；诗=true，杂文=false
  "background": "",                    // 只给读者看的背景小注（写于何时/何境），默认空
  "source": ["device-a"],              // 出处设备，数组；换一个设备导出时追加新的标签
  "guid": "abc123...",                 // 原始设备 guid，仅留档
  "content_hash": "sha1(content)"      // 入库与每次 content 变更时更新（见下）
}
```

## content / note 拆分（关键）

- 默认 `content` = 整条备忘录全文，`note` = 空。
- 有些诗正文后跟着"自注：洋洋洒洒几千字"。**这些必须不进读者视野**——否则自注会淹掉诗、带偏判断。
- 但作者不预先清洗。做法：**GUI 提供"划取 content 里的一段 → 移入 note"的动作**；剪入的部分从诗中收起，折叠成诗下方一个可展开的"自注"小块，**只有作者能展开，读者看不到**。
- 因此 content 的编辑权始终在作者手里、随时可改（见"人改契约"）。

## 读者池规则（visibility × ai_read）

一部作品进入 AI 读者池，当且仅当 **`visibility == "public"` 且 `ai_read == true`**。

- 诗：默认 public + ai_read=true → 被读。
- 作者觉得某首太私人（消沉、需时间才示人）：GUI 里点一下改 `private` → 立刻退出读者池，不再被任何 AI 读、不进任何分享视图。哪天想示人，改回 public 即可——**"需要时间才展示"是一个作者手控的开关，不是一次删改。**
- 杂文：`ai_read=false`（也可视作 archive），永不被读，静态存档，作者自己在 GUI 里慢慢读删。

## 人改 ↔ AI 跑 契约（content_hash）—— FROZEN

作者要能随时手改 corpus，AI 还在后台跑，且互不腐蚀。规则：

- `corpus/诗稿.json` 是**唯一真源**，作者可手改（GUI 或直接编辑）。
- 每条**阅读记录**里存它当时所读版本的 `content_hash`（见 03）。
- 作者改了某首 → 更新该诗 `content_hash`；**旧的阅读记录不删**（历史是历史），但因 hash 不匹配会被自动标为"读的是旧版"。
- 于是：作者 edit 自由、AI 跑 自由、且一眼能看出哪些评论已过时。这正是 WordPress 那类模板给不了的"我读我改、它继续跑、乱不了"。

## 后续（不阻塞 v1）

- 换个设备/来源再导一批时：并入同一 `诗稿.json`，`source` 追加，`id` 入库时统一。多个来源基本无重复，**不写去重脚本**，个别重复由作者手工确认合并。
- genre 重标（如某个分类里混入了别的文体）：只改字段值，文件结构不动。
