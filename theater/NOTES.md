# NOTES —— 实现方（CC / Fable 5）的设计决定与理由

> 按 00 的约定：FROZEN 部分未动过一个字段；以下都是 LATITUDE 范围内的决定，供作者复核。

## 入库（build_corpus_huawei.py）

- **剥标题行**：备忘录导出的正文首行重复标题，入库时剥掉（宽松比较：把不间断空格 \xa0 与全角空格归一后相等才剥）。有少数几首诗的首句本来就与标题相同，各保留了一份——那是诗的第一句，不是导出赘余。
- **id 按 created 时间顺序发号**（zq-0001 = 最早），设备无关，符合 01。
- **date_written 全部初始为 null**：正文里的日期落款五花八门（"完于：2021/8/28"、"11/23 11:32 夜"…），自动抽取风险大于收益；作者可在诗详情页用「写作时间」按钮逐首标注（这也是 GUI 新增的第 4 个动作，01 之外的纯加法）。时间轴在无 date_written 时退回 created，不受影响。
- **genre 原样保留**（现代诗 / 词 / 歌词 / 杂文按原始归类分布）；`ai_read = genre != 杂文`。
- 脚本发现 `诗稿.json` 已存在时拒绝运行——corpus 只进不毁；后续来源到时写增量并入，不覆盖。

## 读者（personas/personas.json，12 位）

设计轴：世代（50后→00后）、母语（1 位英语母语译者）、感受取向（感觉/形式/语言/意象/节奏/结构/脉络/情绪/语言游戏）、knows_诠释（2 位）、knows_date（2 位，其中 1 位与知情重叠成「全知档案员」）。

理由：v1 用一个主模型 × 12 人设即可跑出光谱；「老学究」对词作、「读出声的人」对断行、「译者之眼」对不可译性各有独立的判据，保证分歧是真分歧而非噪声。底线（02 FROZEN）写死在 runner 的 BASELINE system prompt 里，每次阅读都带。

## 跑批（runners/runner.py）

- reads 存储选了 **单一 append-only JSONL**（results/reads/reads.jsonl）：永不覆盖、易 append、覆盖账一次遍历可算。
- 覆盖账是计算视图（coverage 子命令），不单独维护文件。
- plan 挑覆盖最薄的 (诗 × 读者)，并列时**随机打散**（否则字母序靠前的诗/读者永远先被读）。
- ingest 校验冻结 schema：score 越界、model 缺失（出处是命根子）一律拒收。
- 传输层 v1 用 **cc-subagent**：每次阅读 = 一个独立 subagent（上下文只有那一份任务文件），保证盲读纯净。model 如实记录（首轮某首诗全员用了 claude-sonnet-5，其余用 claude-haiku-4.5）。

## 应用（src/server.py + webapp/）

- **零依赖**：Python 标准库 HTTP 服务器 + 原生 JS 前端，作者机器上 `python theater\src\server.py` 即跑，永不烂尾于依赖升级。
- **写 corpus 的唯一通道**是 /api/action 四个作者动作（可见性/剪自注/背景/写作时间）+ /api/promote（诠释升格）；**每次写前自动备份**到 corpus/.backups/（带时间戳，可回滚）。
- **剪自注**的选区映射：浏览器选区丢失空行信息，前端按「选区首行在源文本的位置 → 末行结束」还原成源文本精确子串再提交，服务端再做一次子串校验，两道保险。
- **反应分布图**：SVG 点状直方图，一点 = 一次盲读（0.5 分一档堆叠），红杆标均值；悬停看读者与短评摘要，点击跳到对应评论卡。单系列、单色（青瓷色），按 dataviz 规范不做图例、文字一律墨色。
- **视觉**：暖纸底（#f6f1e7）+ 墨字 + 青瓷点缀（不走全黑，作者唯一硬偏好）；诗体宋体、行高 2.35、最大 34em 居中；评论区与诗之间留 5.5rem 空白——评论不挤诗。
- **榜单**：招牌榜（≥3 读均分）、最两极榜（≥4 读方差）、长/短诗榜、诗词榜；全部前端从盲读记录现算，无任何 LLM 排名（FROZEN 铁律）。「最差榜」按 04 的要求不存在。
- 渲染诗体时对「行行空一行」的导出格式做了自适应（空行占比 ≥70% 判为双倍行距，仅 2 连空行才渲染成节间空），否则正常空行 = 分节。

## 2026-07-11 · 分段侧车 / 古人读者 / 读者墙（第 6 轮补）

- **分段不算修订（corpus/分段.json 侧车）**：查证发现某批原始备忘录导出把每行存成一个段落（多数诗是严格「行_行_行」交替，全库无连续两空行），分段信息在导出时即已丢失。作者手工补分段被定性为**恢复丢失的信息，而非修订文本**——所以存侧车（{诗id: [非空行下标]}），content 与 content_hash 不动，已有阅读记录不标旧版；runner 的 plan 会把分段应用进今后读者读到的正文（顺带把导出的假空行归一为单倍行距）。诚实的代价：分段前后的读者读到的排版不同，这一点记录在此，接受。
- **沈拙庵（清嘉庆落第秀才，通用人设）**：防「一直拿看不懂压分」与「装懂不困惑」的设计——评分尺度写死为三样不随世换的东西（情之真伪/气之贯断/字之警钝），并明令「不识其物不扣分、不许装懂、困惑与臆测本身入批语」；预先告知「此体不叶韵是本然」堵住对诗体本身的系统性压分。虚构人物而非真名人（避免刻板印象与掉书袋）。reads_background=false（背景小注是现代白话，会破坏其知识封界）。**派发建议 sonnet 及以上**（haiku 文言易露馅）。
- **读者墙（#/readers）**：卡片按已读数排序，未上场的归入「候场」区。顶栏加「读者」入口；顶栏改吸顶。
- **最近的评论**：改为整条评语展示、渐进展开（首屏 10 条，每次 +30），供作者批量跑完后在此扫一遍。

## 2026-07-16 · 设置页（corpus/settings.json 侧车）

- **动机**：站点身份（集名/副题/页脚句）与偏好散落在 index.html / server.py 硬编码里，开源后新用户改不掉；派发偏好（默认模型/层数）只活在 SKILL.md 文字里，没有机器可读的落点。设置页把两类都收口到 `corpus/settings.json` 一个侧车 + `/api/settings` 一个端点，**GUI 与派发 agent 读写同一份文件**。
- **只存差异**：文件里只落作者显式设置过的项，空字符串/null = 从文件删掉该项、回默认（`DEFAULT_SETTINGS` 在 server.py）——默认值永不固化进用户文件，将来改默认不会被旧文件钉死。全清空时文件本身删除。白名单校验：`default_view`/`score_badge` 枚举、`port`/`target_depth` 整数区间，未知字段一律丢弃。
- **路由拆分**：`#/` 从"榜单的家"改为"作者偏好的家"（default_view），榜单自己搬去 `#/boards`——否则顶栏「榜单」和默认页共用 `#/` 会互相打架。
- **评分口径开关**（score_badge=raw）只在前端 `stats()` 一个咽喉处生效：cal 置 null，全站（榜单排序/徽章/众目）自动回退均分；统计页自己的原始/校准开关独立不受影响。
- **收进来的**：集名/副题/页脚句/默认页/评分口径/端口/派发三项。**明确不收**：主题色（SVG 图表颜色散落 JS 十几处，暖纸+青瓷是设计资产）、作者署名（那是 corpus 数据不是偏好）、榜单读数阈值（统计学参数不该让用户拧）。

## 2026-07-16 · 多文体阅读（v1.3）

- **换判据不换人**：非诗文体不做新读者、不写新底线——读者 prompt 在诗歌底线后追加「体裁转换」段：声明"这不是诗，是一篇 X"、四条底线原样适用、诗的判据换成该文体的对应物、不必更苛刻也不必更宽容（作者定的形态：同样欣赏的眼光，按各自性格读）。12 位读者的性格是资产，跨文体复用。
- **手动勾选才读**：文体门收口到设置页 `read_genres`——诗类（现代诗/词/歌词）永远在池，其他文体作者勾选才进；GUI 改文体时非诗一律 ai_read=false（原先只有杂文/草稿退出，自定义文体如"剧本"会带着诗歌标准被读，正是本轮堵的漏洞）。`genre_notes` 让作者给每个文体写一两句自己的评判要求附进 prompt；「添加文体」一个输入框即完成新文体接入——这就是全部框架，刻意不做更复杂的。
- **校准池按"标准"分界，不按作品当前文体**：STANDARD_SPLIT=2026-07-17。此前全部读数（含相当一部分落在后来被作者改标为杂文/草稿的作品上的）都产生于诗歌底线之下，是合法的 rater 松紧原料，全部保留（校准零漂移）；此后的非诗读数产生于转换标准，不进诗的校准池，相应作品前端自动回退原始均分（现成兜底逻辑，零改动）。某文体读数攒够之前不建独立校准池。

## 2026-07-17 · Persona 外挂（侧车持久化）

- **默认与自建分开，一起管理**：随附人设留在 git 跟踪的 `theater/personas/personas.json`（随更新可覆盖）；读者自建/改写放侧车 `corpus/personas.json`（`corpus/` 已 gitignore，`git pull` 永不覆盖）。`load_personas()` 按 `persona_id` 合并——同 id **部分覆盖**（只改给出的字段，其余保留随附值）、新 id **追加**、`hidden:true` **撤下**某随附。无侧车时与旧版字节级一致。server `/api/state`、runner `cmd_plan`、`plan_thicken` 三处统一走它，口径一致。
- **岔路 B（改写随附 id 的隐患）决策**：新 id = 纯追加、绝对安全；改写随附 id 的人设文本**允许但不硬拦**——会让挂着同一 id 的旧读数语义漂移（读的其实是改前的人设），因为那是读者自己的库，知情即可，不做校验。
- **写 API**：`/api/personas` POST 整份替换侧车（`set_personas`，白名单校验、原子写、空列表删文件），照抄 `set_settings` 范式；全新 id 必须含 name+persona（否则无法展示/派发），`knows_*` 缺省补 False。读者也可直接手改 `corpus/personas.json`（复制 `theater/personas/personas.sidecar.example.json` 起步）。
- **读者页 GUI（同日补齐）**：读者墙底部单开「你的读者」区（自建的强分区展示，徽标「你的」＝存在侧车、更新不覆盖；改写过的随附读者标「已改写」）＋「＋ 新建读者」；点进任何读者页可「编辑 / 删除(自建) / 撤下(随附) / 还原(改写)」，删除撤下走两击确认。编辑随附读者时只把**与随附值不同的字段**写进侧车（部分覆盖语义在 GUI 里保真：没改的字段继续跟随更新），零差异不留空壳条目；表单里明示岔路 B 的语义漂移。为此 `/api/state` 增发 `personas_defaults` 与 `personas_sidecar` 两键（`/api/personas` 是整份替换，前端必须拿到侧车全份才能增删改）。「联网共享人设 / 创意工坊」仍是更远的将来项。

## 非诗文体榜（2026-07-17）

- 非诗作品 `ai_read=false`、不进 `pool()`、也不进诗的校准（判据不同、同榜不公平）——但此前它们**没有任何榜**，读者反馈"找不到非诗的榜"。新增：榜单页「更多榜单 · 非诗文体」按各自文体的**原始均分**单独排（≥1 读即上榜），与诗榜彻底分离，不共享校准量表。某文体读数攒够后再考虑独立校准/完整榜页。

## 2026-07-17 · 校准偏差感知收缩（trust）

- **问题**：等分位映射的前提是各模型面对的诗池质量可比。低读数模型（opus/fable，各约 10 读）被派的全是好诗——诗池代理偏离 +0.86/+0.83（其余模型都在 ±0.2 内），旧算法把"读的全是好诗"误译成"这个 rater 手松"，把它们打出的高分整体压低，恰好惩罚了被它们读过的诗。
- **解法**：均衡诊断的 `proxy_dev`（leave-one-model-out 诗池代理偏分）直接变成信任因子 `trust = max(0, 1-|dev|/D_TRUST)`（D_TRUST=0.6），乘进该模型自身分布的收缩权重 `w_cell`/`w_model`。诗池失衡的模型自动滑向全局先验、近似恒等映射；均衡的模型基本不动（trust 0.82–0.95，换算漂移 ≤0.1）。不做"忽略高于均值的修正"这类单侧规则——那会破坏等值的对称性。
- **效果**（改后首跑）：opus/fable 换算变为近恒等（7.0→7.00、8.0→8.00，旧版 7.5→7.3/7.4）；被它们读过的两首高覆盖诗恢复应有名次（8→3、15→4）。诊断本身零漂移，警告仍保留（提醒检查派发，trust 只是兜底不是赦免）。参考分布未动（不 refreeze）。报告的均衡诊断表加「信任」列，scores.json meta 加 `d_trust` 审计字段。

## 2026-07-17 · 跟帖模式（thread）设计定稿（施工前，Opus/Fable 5 交叉设计）

**动机**：`context_mode: thread` 早在 03 的 schema 里就留了钩子（`thread_ref` 指向被回复的 `read_id`），但只是"以后再说"。这轮把它从钩子变成可以动手写的设计——不是给盲读提速，是给少数几首已经有"长评"（深读）的诗办一场"读者茶话会"：看读者放在一起会不会吵起来、会不会被说服、会不会在别人观点上看出新东西。这是和盲读完全不同类型的信号，数据永远分开存、不进任何统计/校准（`calibrate.py` 本来就按 `context_mode == "blind"` 过滤，天然满足）。

**可见范围——祖先链 ∪ 自己历史，不是全楼也不是随便开的滑动窗口**：
- 读者接的是某条具体楼层（`thread_ref` 指向单个父楼层，本来就是树状不是扁平），看得到的是这条楼层一路往上追溯到楼主帖的祖先链，外加**该读者自己在本帖已经发过的楼层**（防止同一读者在不同分支里自我打脸——这是纯技术漏洞，早期方案没考虑到，看自己说过的不算被别人带偏，不违反隔离原则）。
- 深度/成本不靠"层数硬封顶"（会把讨论压成"表态陈列"，杀死"第5轮之后才会发生的说服弧线"），改按**祖先链原文的 token 预算封顶**；要不要上摘要压缩、具体预算数字，都不预先拍——先挑 2 首诗试跑，肉眼看第几层开始复读/跑题，退化点前一层就是答案。这也是本项目一贯的脾气：不做"自动判断可能悄悄出错还没法肉眼核验"的机制，宁可先空跑一次看现实什么样。

**v0 先用派发方指定接楼**，不做"读者自己挑楼层"（那需要读者先看全楼才能选，选择那一刻污染已经发生，等于自欺欺人）。读者自选作为 v1.1 的候选，等 v0 判断出"讨论会不会真分裂"这件事本身跑不跑得通再加。

**楼主（长评作者）默认享有优先回护自己观点的权利**——开楼帖必须绑定在已有"长评"上而非随便一条盲读短评，这样才有实质内容值得别人接话，也顺带限定了"只有精选诗定点开"。

**立场惯性靠人格真实审美先验，不靠"别轻易被说服"这类指令**：那类指令暴露了期望方向，会被表演（讨好式附和 → 讨好式装作难被说服，同样是演）。正解是把"你判断诗先看什么、什么理由动不了你"写进人格文件本身——**上线前要过一遍现有 personas.json，缺"审美优先级"颗粒度的读者补一两句**，这是小增量，不是新机制。兜底仍留一条对称提示（改立场/不改立场都要出示机制，不提哪边更好），防止彻底没有下限。

**沉默是完整产出，不是"允许但没人选"的摆设**：举证责任反过来——开口需要理由（"这层楼有没有一个具体的、楼里还没人说过、你能一句话说清的点"），沉默也要写一句原因、同样存档，不是空手而归。不写具体比例（会变成模型表演性凑配额）。真正的稀疏度靠**派发层**兜底：每层楼只邀请一部分读者（如随机抽一半），两层叠加才是自然的热闹程度。连续两层无人接话 → 封帖（终止条件，具体阈值留给试点校准）。

**复述/引用格式只做内部脚手架，不进最终展示**：生成时要求【接住的原句】（逐字引用父楼层的承重句）【我的转述】【回应】三段，但只有【回应】（去标签、自然行文）存进正式记录——各位读者的声音不能被同一套标签磨平。【接住的原句】用于派发方做字符串匹配校验，引用对不上就**静默重roll**（此时还没有任何记录落盘，不涉及 append-only 或 void 问题）。

**格式对了但内容垮了（事后才看出人格崩坏/跑题）→ 标 `void`，不删除、不改写**，参考 `curation.json`（折叠评论）的先例：隐藏但不销毁底层记录。

**新增元数据不进 FROZEN 的 reads.jsonl 字段**（`persona_hash`、链深、是否声明立场变化、是否选择沉默这几项），照抄 `分段.json`/`settings.json`/`personas.json`/`curation.json` 的先例，另开一个 thread 专属侧车存，reads 的字段集一个不动。`persona_hash` 做法完全比照 `content_hash`：读的时候顺手记一份人格文本的 sha1，人格改了就知道读的是哪版。

**派发顺序随机化并记录**——复用 03 里"并列时随机打散"的既有原则，不是新规矩。

**已知开销**：这套设计比 blind 复杂得多，符合预期——thread 本来就是"偶尔办的沙龙"，不追求跟 blind 一样能无限堆量。
